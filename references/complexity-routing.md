# 复杂度分流参考

## 目的

用统一标准判断这次请求该走多强的流程。

## 三档复杂度

### `light`

适用情况：

- 目标已基本清楚
- 范围稳定
- 只需要轻澄清、轻规划或看当前态势

默认处理：

- 给最小可执行结果
- 不强制展开完整闭环

### `standard`

适用情况：

- 需要完整推进一个目标
- 需要经过目标、计划、执行、验收

默认处理：

- 进入 `want -> plan -> todo -> achieve | abandon`

### `deep`

适用情况：

- 多目标冲突
- 长期推进
- 阻塞复杂
- 验收口径不稳定
- 需要复盘或知识治理

默认处理：

- 在标准闭环基础上，补事实、风险、证据、缺口
- 必要时转 `reflect` 或 `wiki`

## 判断维度

优先看四件事：

1. 目标是否清楚
2. 范围是否稳定
3. 是否涉及长期状态维护
4. 是否需要知识沉淀或复盘

## 升降级规则

满足以下情况之一时升级：

- 目标边界明显变化
- 需要跨多个阶段连续推进
- 阻塞已经影响下一步判断
- 验收标准明显不清
- 本轮必须引入 `reflect` 或 `wiki`

满足以下情况之一时可降级：

- 用户只要当前态势
- 目标已清楚，只缺一个最小下一步
- 本轮不需要落盘、验收或沉淀

## 动作关系

- `focus` 常见于 `light`
- `want`、`plan` 常见于 `light` 或 `standard`
- `todo`、`achieve` 常见于 `standard`
- `reflect`、`wiki` 更常见于 `deep`

注意：

- `focus` 只负责观察，不等于低复杂度
- `todo` 负责执行，不等于一定复杂
