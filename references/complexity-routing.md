# 复杂度分流参考

## 目的

为 `heroagent` 提供统一的复杂度分流规则，避免所有请求都走同一套流程强度。

## 总体规则

先判断复杂度，再决定流程强度。  
复杂度不是按文字长短判断，而是按“需要多强的治理动作”判断。

## 三档复杂度

### `light`

适用情况：

- 目标已基本清楚
- 范围稳定
- 当前只需要轻澄清、轻规划或快速看态势
- 不强制要求完整落盘

默认动作：

- `want`
- `plan`
- `focus`

默认行为：

- 优先给最小可执行结果
- 不自动展开完整闭环
- 不因为信息稍多就升级

### `standard`

适用情况：

- 需要明确目标
- 需要拆阶段和任务
- 需要推进并验收

默认动作：

- `want -> plan -> todo -> achieve | abandon`

默认行为：

- 走标准推进闭环
- 默认连续执行 `todo`
- 验收后再决定归档或补缺口

### `deep`

适用情况：

- 多目标冲突
- 长期推进
- 阻塞复杂
- 验收口径不稳定
- 需要复盘或知识治理

默认动作：

- 在标准闭环基础上，必要时转入 `reflect` 或 `wiki`

默认行为：

- 强制补风险、证据和缺口
- 明确区分事实、判断、待确认
- 不允许在证据不足时强行收口

## 判断维度

优先看四件事：

1. 目标是否清楚
2. 范围是否稳定
3. 是否涉及长期状态维护
4. 是否需要知识沉淀或复盘

## 升级规则

满足以下情况之一，可从 `light` 升到 `standard`，或从 `standard` 升到 `deep`：

- 用户目标边界发生明显变化
- 需要跨多个阶段连续推进
- 阻塞已经影响下一步判断
- 验收标准明显不清
- 本轮必须引入 `reflect` 或 `wiki`

## 降级规则

满足以下情况，可降低流程强度：

- 用户只需要当前态势，不需要继续规划
- 目标已经清楚，只需要一个最小下一步
- 本轮不需要落盘、不需要验收、不需要沉淀

## 与动作的关系

- `focus` 常见于 `light`
- `want`、`plan` 常见于 `light` 或 `standard`
- `todo`、`achieve` 常见于 `standard`
- `reflect`、`wiki` 更常见于 `deep`

注意：

- `focus` 是观察动作，不代表复杂度低
- `reflect` 也不一定总是 `deep`，但一旦涉及根因确认和经验沉淀，通常应按 `deep` 对待
