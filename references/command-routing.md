# 指令路由参考

## 目的

把用户的自然语言请求稳定映射到 `heroagent` 的公开动作，并在需要时选择合适的内部沉淀方法。

## 总体规则

- 先判断用户是否使用了显式指令 `~动作`
- 先判断用户是否在要求初始化工作区
- 先判断用户是在推进目标、复盘问题，还是维护知识
- 再判断当前最阻塞的一步是什么
- 若一句话同时命中多个动作，先处理前置动作，再处理后置动作
- 若用户只描述现象，没有明确诉求，优先补齐最小上下文后再路由

## 显式指令优先级

当用户输入以下任一指令时，直接命中对应动作，不再做自然语言推断：

- `~init`
- `~wiki`
- `~want`
- `~plan`
- `~todo`
- `~focus`
- `~achieve`
- `~abandon`
- `~reflect`

## 初始化动作路由

### `init`

当用户要求初始化 `.heroagent/`、接管老项目并开始长期记录、或希望自动创建工作区时，路由到 `init`。

## 知识动作路由

### `wiki`

当用户要求维护知识库、更新项目概览、补充架构/API/数据说明，或希望优先利用现有知识作为上下文时，路由到 `wiki`。

## 推进闭环路由

### `want`

当用户在表达愿望、方向、想法，但还没有形成可验收目标时，路由到 `want`。

不要误判为 `plan` 的情况：

- 用户已经给出明确成功标准和边界
- 用户真正缺的是阶段路径，而不是目标定义

### `plan`

当用户目标已明确，但不知道如何分阶段推进时，路由到 `plan`。

不要误判为 `todo` 的情况：

- 用户要的是阶段，不是零碎任务
- 当前需要先明确关键路径，而不是直接列清单

### `todo`

当用户已经有目标或计划，需要形成可执行清单时，路由到 `todo`。

不要误判为 `focus` 的情况：

- 用户不是在问现状，而是在要执行项

### `focus`

当用户在问当前状态、优先级、卡点、下一步时，路由到 `focus`。

补充规则：

- `focus` 是观察动作，不属于主流程阶段
- 若用户只是在同步“这批任务做完了”，但还没要求验收，先更新完成状态，再按上下文进入 `focus` 或 `achieve`

### `achieve`

当用户要求验收目标是否真正闭环时，路由到 `achieve`。

补充规则：

- 若用户在反馈任务完成的同时询问“现在算完成了吗”“能不能收口”，直接路由到 `achieve`
- 进入 `achieve` 前，应先把当前状态更新为“已完成，待验收”

### `abandon`

当用户决定停止推进、暂缓、砍掉某个目标时，路由到 `abandon`。

不要把 `abandon` 包装成 `plan`：

- 明确停止时，应直接做停损和收口

## 复盘与知识维护

### `reflect`

当用户要复盘原因、问题、失误、延期、偏差时，路由到 `reflect`。

### `wiki`

当用户要压缩长记录、整理历史变更、提炼一句话经验、清理旧知识或淘汰失效约束时，公开路由到 `wiki`。

内部处理规则：

- 复盘结论被用户确认后，可内部触发 `realize`
- `wiki` 维护时，可内部选择 `synthesize` 或 `forget`
- 当某条经验已上升为稳定执行标准时，再内部触发 `master`

## 多动作组合路由

### 组合一

“帮我定目标，再拆计划，再列 todo。”

路由顺序：

1. `want`
2. `plan`
3. `todo`

### 组合二

“这个任务做完了，帮我看现在是否能收口。”

路由顺序：

1. 更新完成状态
2. `achieve`

### 组合三

“这次延期了，帮我复盘并总结以后怎么做。”

路由顺序：

1. `reflect`
2. 用户确认复盘结论
3. 内部执行 `realize`
4. 必要时内部执行 `master`

## 路由失败时的兜底

如果无法稳定判断动作，先补一个最小澄清问题，优先问：

- 你现在更需要我帮你定目标、拆计划，还是复盘总结
- 你要解决的是推进问题，还是复盘问题
- 你现在最缺的是下一步动作，还是对结果的判断

避免一口气给多个大而散的选项。
