# 输出规范

## 目的

作为 `heroagent` 的统一输出规范源，定义所有动作下的固定输出结构、字段顺序和质量要求，保证输出可复用、可检查、可衔接下一步。

## 通用规范

所有动作输出都应满足以下要求：

1. 第一部分直接给结论
2. 第一行使用单行状态头
3. 开头必须先给 `重点摘要`
4. 字段名稳定，尽量不要同义改写
5. 缺失信息明确标注，不伪造内容
6. 最后给出 `下一步`

## 人类优先规则

- 默认把“人类先看什么”放在最前面，而不是把完整分析放在最前面
- 单次回复优先服务“继续推进”而不是“展示思考过程”
- 若详细说明较长，必须先给 1 到 3 条重点摘要，再展开
- 若存在需要用户决策的内容，优先压缩成低成本选择，不要要求用户读长文后自行总结
- 若用户没有回应某个选择，默认视为“暂不处理”，不要反复追问

## 重点摘要

所有需要人类阅读的正式输出，建议在状态头后优先给：

```md
重点摘要：
- ...
- ...
```

质量要求：

- 控制在 1 到 3 条
- 每条只保留最关键结论
- 不重复后文字段名

## 选择格式

当需要用户做决定时，统一优先使用轻量选择格式：

```md
请直接回复：
1. ...
2. ...
3. 暂不处理
```

规则：

- 优先提供 2 到 3 个选项
- 若是二选一，也必须加上 `3. 暂不处理`
- 用户若不回复或避而不谈，视为 `暂不处理`
- 不要把多个不同维度的问题混在同一组选项里

## 顶层输出结构

统一使用以下结构：

```md
{状态符号} HeroAgent · {场景名称}

重点摘要：
- ...

[主体内容]

下一步：
- ...
```

## 状态符号

- `✅`：已完成
- `❓`：需要用户输入或确认
- `⚠️`：存在风险、缺口或部分完成
- `🚫`：用户取消或已终止
- `❌`：执行错误
- `💡`：纯问答或轻咨询

## 场景名称

推荐使用：

- 初始化
- 目标定义
- 计划制定
- 任务拆解
- 进度聚焦
- 任务完成
- 目标验收
- 放弃收口
- 问题复盘
- 原则提炼
- 流程沉淀
- 知识传授
- 规则淘汰
- 咨询问答
- 需要确认
- 执行错误

## 通用字段规则

- 所有字段优先使用中文固定字段名
- 同一动作不要混用近义字段
- 若字段无内容，可留空，不要删字段名
- 若需要落盘，优先与 `assets/templates/` 保持一致
- emoji 用于状态和重点提示，不要堆砌装饰性符号

推荐通用收尾字段：

- `下一步`：最合理的后续动作
- `请直接回复`：当确实需要用户选择时再出现

## 动作规范

### `init`

必备字段：

- `工作区`
- `初始化结果`
- `已创建`
- `下一步`

质量要求：

- `初始化结果` 只写 `已初始化`、`已补齐`、`无需变更`
- `已创建` 应列出关键目录或关键文件
- 初始化后必须给出一个明确的后续动作
- 若需要用户决定是否继续生成首批目标，应用选择格式给出 `继续生成`、`仅初始化`、`暂不处理`

### `wiki`

必备字段：

- `知识范围`
- `更新结果`
- `已读取或已更新`
- `建议同步`
- `下一步`

质量要求：

- `知识范围` 应明确指向 `overview`、`arch`、`api`、`data` 或模块级 wiki
- `更新结果` 只写 `已更新`、`已补齐`、`已读取`、`无需变更`
- 若本次是知识消费而非知识维护，也应说明消费了哪些 wiki 文件
- 若本次基于代码变更推导 wiki，同步建议应显式列出建议刷新的 wiki 文件
- 若本次写入模块级 wiki，`知识范围` 应明确到模块名，例如 `modules/payments.md`
- 若本次刷新了索引或注册表，也应说明 `index.md`、`registry.json` 是否同步完成
- 若本次只生成了待补写草稿，应显式说明草稿位置，并标注“未直接写入正式 wiki”
- 若本次已应用草稿，应显式说明目标 wiki、草稿是否删除，以及索引是否已刷新
- 若本次输出维护报告，应显式说明报告位置，以及缺失草稿、陈旧草稿、可合并草稿三类结果
- 若本次执行自动收敛，应显式说明自动创建了哪些草稿、合并了哪些 wiki、标记了哪些陈旧草稿
- 若本次执行压缩，应显式说明压缩了哪些 wiki 文件，以及是否生成了 `自动同步摘要`
- 若本次输出信号评分，应显式说明评分最高的 wiki 及其 `priority`、`freshness`、`density`、`draft_dependency`
- 若本次装配上下文包，应显式说明当前动作、入选 wiki 文件以及装配理由
- 若本次执行源码提炼，应显式说明提炼到了哪些目标 wiki，以及生成了哪些 `facts__*.md`
- 若本次执行维护策略，应显式说明采用了哪种策略，以及自动执行了哪些动作
- 若需要用户决定是否继续补写、合并、压缩或切换策略，必须使用选择格式，避免开放式追问

### `want`

必备字段：

- `目标`
- `成功标准`
- `范围`
- `不做什么`
- `约束`

质量要求：

- `目标` 必须可判断是否达成
- `成功标准` 不能只是“做好”“优化”
- `不做什么` 必须能收边界
- `约束` 至少覆盖时间、资源、技术或业务限制中的关键项
- `背景`、`价值` 作为可选补充，不属于最小目标卡必备字段
- `下一步` 应放在回复层、`current-focus.md` 或 `workflow-state.json`，不写入目标卡本体
- 在正式输出目标卡片前，先完成需求完整度评分
- 评分 `< 8` 时，不输出正式目标卡片，改用 `目标澄清` 输出

### `want` 评分不足时的澄清输出

必备字段：

- `当前评分`
- `缺失信息`
- `当前问题`
- `下一步`

质量要求：

- `当前评分` 使用 10 分制
- 每一轮只允许 1 个 `当前问题`
- `当前问题` 必须是最关键的收敛问题
- 允许用户回复“按现有信息继续”
- 不允许 AI 在 `want` 阶段替用户做关键取舍
- 达到可进入下一步时，必须提供：
  - `1. 继续下一步 ~plan`
  - `2. 继续补充需求`
  - `3. 暂不处理`
- 应同步把当前状态写入 `progress/workflow-state.json`

### `plan`

必备字段：

- `阶段`
- `产出`
- `风险`
- `完成标志`

质量要求：

- 至少能看出先后顺序
- 每个阶段都要有清晰产出
- `完成标志` 必须可验证
- 若阶段较多，重点摘要只保留关键路径和最近一步

### `todo`

必备字段：

- `任务名`
- `完成定义`
- `依赖`
- `优先级`

质量要求：

- `任务名` 用动作开头
- `完成定义` 要能判定 done
- `依赖` 为空时也可写“无”
- 若任务超过 5 项，正文可展开，但重点摘要只保留最优先的 1 到 3 项
- 默认执行策略是按优先级连续推进全部任务，不逐项请求用户确认
- 只有遇到关键取舍、高风险操作、权限问题或外部阻塞时，才允许中断并请求确认
- `todo` 全部完成后，应自动更新为“已完成，待验收”状态，并将下一步切到 `achieve`

### `focus`

必备字段：

- `当前目标`
- `当前阶段`
- `已完成`
- `进行中`
- `阻塞`
- `下一步`

质量要求：

- 不复述无关历史
- `阻塞` 必须具体，不写空泛风险
- `下一步` 只能给一个最优先动作
- 重点摘要应优先写“当前最该做什么”

### `achieve`

必备字段：

- `目标`
- `验收结果`
- `证据`
- `剩余缺口`
- `是否归档`
- `下一步`

质量要求：

- `验收结果` 只使用 `已达成`、`部分达成`、`未达成`
- `部分达成` 时必须写清缺口
- `是否归档` 不能脱离证据单独判断
- 归档判断只能出现在 `achieve`，不能在任务完成后提前给出
- 若需要决定是否归档，应用选择格式给出 `归档`、`继续补缺口`、`暂不处理`

### `abandon`

必备字段：

- `放弃目标`
- `放弃原因`
- `已投入沉没成本`
- `可保留资产`
- `重启条件`
- `下一步`

质量要求：

- `放弃原因` 要落到价值、时机、资源或方向变化
- `可保留资产` 优先写还能复用的内容
- 若用户表达犹豫，应用选择格式给出 `继续推进`、`正式放弃`、`暂不处理`

### `reflect`

必备字段：

- `现象`
- `影响`
- `根因`
- `当时漏判了什么`
- `下次如何更早发现`
- `下一步`

质量要求：

- `根因` 不能停在表面症状
- `下次如何更早发现` 必须包含观察信号或检查动作
- 重点摘要优先给“根因一句话”和“下一次最该提前检查什么”

### `realize`

必备字段：

- `原则`
- `触发信号`
- `适用边界`
- `推荐动作`
- `反例`
- `下一步`

质量要求：

- `原则` 应该可迁移
- `适用边界` 要说明何时不该用
- `反例` 用于防止过度泛化

### `master`

必备字段：

- `流程名`
- `适用场景`
- `入口条件`
- `步骤`
- `出口条件`
- `常见失败点`
- `下一步`

质量要求：

- `步骤` 必须有顺序
- `入口条件` 与 `出口条件` 都要可判断
- `常见失败点` 要能指导执行避坑

### `synthesize`

必备字段：

- `教什么`
- `适合对象`
- `核心模型`
- `示例`
- `常见误解`
- `一句话带走`
- `下一步`

质量要求：

- `核心模型` 要足够简洁
- `示例` 优先真实、最小、可复述
- `一句话带走` 要能单独成立

### `forget`

必备字段：

- `淘汰对象`
- `过时原因`
- `失效信号`
- `替代认知或替代流程`
- `迁移建议`
- `下一步`

质量要求：

- `过时原因` 必须说明为什么现在不成立
- `替代认知或替代流程` 不能为空
- `迁移建议` 应可执行，而不是一句“以后不要用了”

## 使用规则

- `SKILL.md` 中涉及输出时，以本文件为准
- `prompt-patterns.md` 只负责提问策略，不重复定义输出字段
- `assets/templates/` 负责落盘模板，应尽量与本文件保持同构
- 正常完成、交互确认、异常中断、轻问答都应使用同一套顶层状态头
