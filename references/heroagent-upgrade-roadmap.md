# HeroAgent 升级清单

## 结论

`helloagents_v1` 最值得学习的，不是它写得更长，而是它把 **顶层路由、阶段职责、模板资产、示例资产** 明确拆开了。

但 `helloagents` 自己后续已经演进到 `v2`，说明 `v1` 只是中间态，不适合直接照搬。  
对 HeroAgent 更合适的升级方向是：

- 保留当前以 `目标推进 + 知识沉淀` 为中心的产品定位
- 吸收 `v1` 的目录分层和模板治理能力
- 避免回退到 `v1` 那种“阶段很清楚，但整体过重、规则堆叠偏多”的形态
- 优先走一条更接近 `v2` 的路线：主入口保持简洁，细则继续下沉，但对外结构更产品化、更可安装、更易理解

---

## 一、对方项目的精华

### 1. 顶层入口和子模块职责清楚

`helloagents_v1` 采用的是：

- 顶层 `SKILL.md` 负责路由和全局规则
- `analyze/`、`design/`、`develop/`、`kb/` 负责阶段细则
- `templates/` 负责模板
- `examples/` 负责示例

这种结构的价值是：

- 用户和模型都容易理解“先去哪一层找规则”
- 顶层说明不会被阶段细节淹没
- 模板、示例、规则职责分离，后续维护成本更低

### 2. 模板体系完整

`helloagents_v1/templates/` 把方案包、知识库、输出格式都拆成独立模板文件。  
这比把模板内容直接写在 `SKILL.md` 里更稳定，也更利于复用。

### 3. 示例资产单独存在

`examples/` 目录把典型路径单独存放。  
这对于统一风格、做回归对照、帮助后来维护者理解“理想输出长什么样”很有价值。

### 4. 路由是面向任务阶段设计的

`v1` 的核心是“需求分析 → 方案设计 → 开发实施 → 知识库同步”。  
虽然这更偏编码场景，但它在工程上有一个优点：阶段切分天然适合做边界控制、模板约束和产物归档。

---

## 二、HeroAgent 当前的优势

相较 `helloagents_v1`，HeroAgent 已经有几项明显优势。

### 1. 产品定位更聚焦

HeroAgent 不是泛化的“开发全流程 skill”，而是明确聚焦：

- 目标澄清
- 路径拆解
- 任务推进
- 当前聚焦
- 结果验收
- 问题复盘
- wiki 与原则沉淀

这个定位更清楚，也更有辨识度。

### 2. 主文档已经比对方克制

当前 [`SKILL.md`](./SKILL.md) 约 222 行，而 `helloagents_v1/SKILL.md` 约 1130 行，`helloagents_v2/SKILL.md` 约 1514 行。  
这说明 HeroAgent 在“高信号入口”这件事上，方向是对的。

### 3. references 体系已经比较成熟

当前仓库已经有：

- 路由矩阵
- 状态管理
- 输出契约
- wiki 消费与同步规则
- requirement scoring
- prompt patterns

这意味着 HeroAgent 并不缺规则，只是还缺一个更清晰的“产品化外壳”和“分层导航结构”。

### 4. 脚本能力比文档表达更成熟

当前 `scripts/` 已经覆盖：

- 初始化
- 首目标引导
- focus 更新
- wiki 草稿闭环
- wiki 信号评分
- wiki 上下文装配
- 归档与巡检

也就是说，HeroAgent 的问题不在能力不足，而在于：

- 能力地图没有被更好地包装出来
- 目录分层还没有把这些能力组织成更易理解的结构

## 三、核心差距

### P1 结构差距

HeroAgent 目前主要是：

- 一个顶层 `SKILL.md`
- 一组 `references/`
- 一组 `assets/templates/`
- 一组 `scripts/`

这种结构对维护者友好，但对首次接触者不够“模块化可理解”。

### P1 心智模型差距

`helloagents_v1` 的目录命名直接映射阶段。  
HeroAgent 的动作模型已经很清楚，但这些动作仍主要集中在主文档和 reference 中，没有形成“公开动作层”和“内部能力层”的目录分离。

### P2 模板治理差距

HeroAgent 目前只有 7 个工作流模板，偏结果卡片。  
相比之下，对方有更完整的：

- 输出格式模板
- 方案包模板
- wiki 模板
- 变更记录模板

HeroAgent 在“长期推进”上更强，但在“标准产物模板化”上还可以继续补强。

### P2 示例治理差距

当前仓库缺少独立 `examples/`。  
这会导致两类成本升高：

- 新维护者难以快速理解理想交互和产物形态
- 未来重构后很难做“文档行为回归”

### P2 对外包装差距

当前 [`README.md`](./README.md) 已经能说明用途，但仍偏说明书。  
相比 `helloagents`，HeroAgent 还可以进一步提升：

- 安装后的第一印象
- 使用路径的最小闭环展示
- 面向不同用户场景的入口设计

## 四、升级原则

升级 HeroAgent 时，建议坚持四条原则：

### 1. 不丢定位

不要把 HeroAgent 改造成另一个通用编码代理。  
它的核心价值仍应是“目标推进与知识沉淀”，而不是“从开发到回滚的全生命周期命令系统”。

### 2. 学结构，不学堆叠

学习 `helloagents_v1` 的模块化组织方式，  
不要学习它把大量全局规则、格式要求、分支条件堆在主文档里的写法。

### 3. 对外更清楚，对内更按需

对用户和维护者要更好理解：目录更清晰、入口更清楚、示例更直观。  
对模型执行仍然保持当前思路：按需读取，而不是全量加载。

### 4. 规则与产物分开治理

应把下面几层明确分开：

- 入口与路由
- 动作细则
- 输出模板
- 示例
- 脚本
- 验证

---

## 五、升级清单

以下清单按优先级排序。

### 第一层 必须做

#### 1. 重构目录为“入口层 + 动作层 + 资产层”

建议目标结构：

```text
heroagent/
├── SKILL.md
├── README.md
├── actions/
│   ├── want.md
│   ├── plan.md
│   ├── todo.md
│   ├── focus.md
│   ├── achieve.md
│   ├── abandon.md
│   ├── wiki.md
│   └── reflect.md
├── references/
├── assets/
│   ├── templates/
│   └── examples/
├── scripts/
├── tests/
└── agents/
```

目标：

- `SKILL.md` 只保留定位、路由、核心约束、按需加载导航
- 每个公开动作各有单独文档
- `references/` 继续承载通用规则
- `assets/examples/` 承载真实示例

这是本轮最重要的升级项。

#### 2. 把动作细则从主文档继续下沉

建议从 [`SKILL.md`](./SKILL.md) 中继续下沉以下内容到 `actions/`：

- `want` 的评分门槛与分支
- `wiki` 的消费与同步门槛
- `todo` 的默认执行策略
- `reflect` 与 `realize/master` 的联动边界
- `achieve` 与 `abandon` 的验收和收口标准

这样做的收益是：

- 主入口更短
- 动作职责更清楚
- 修改单个动作时不必影响主文档

#### 3. 新增 examples 目录

建议新增至少 6 个示例：

- `want-example.md`
- `plan-example.md`
- `todo-example.md`
- `focus-example.md`
- `reflect-example.md`
- `wiki-sync-example.md`

每个示例只回答一个问题：

- 典型输入是什么
- 理想输出长什么样
- 何时应落盘
- 何时应停止追问

#### 4. 建立文档一致性检查

当前优化日志已经指出缺少文档一致性测试。  
这项应尽快补上，至少覆盖：

- `SKILL.md` 中引用的 references 文件必须存在
- `README.md` 中列出的脚本必须真实存在
- `SKILL.md` 中列出的模板必须真实存在
- `actions/` 中引用的脚本与 references 必须可解析

建议新增一个轻量测试，例如：

- `tests/test_docs_consistency.py`

这是把 skill 从“能看”升级到“可持续维护”的关键一步。

---

### 第二层 应该做

#### 5. 补齐动作级模板

当前模板偏“卡片结果”。建议新增：

- `want-clarification.md`
- `plan-outline.md`
- `achieve-checklist.md`
- `abandon-summary.md`
- `wiki-draft.md`

这样可以让 HeroAgent 的产物从“有结果”升级为“结果更稳定”。

#### 6. README 改成外部用户视角

建议把 [`README.md`](./README.md) 进一步压缩成三块：

- HeroAgent 是什么
- 什么时候该用
- 一个最小完整工作流

脚本清单、深入规则、wiki 机制不应在 README 里铺太多，统一引到对应文档。

#### 7. 建立“公开动作”和“内部机制”的边界文档

HeroAgent 当前有公开动作，也有内部机制：

- 公开动作：`init`、`wiki`、`want`、`plan`、`todo`、`focus`、`achieve`、`abandon`、`reflect`
- 内部机制：`realize`、`synthesize`、`forget`、`master`

建议单独补一份边界说明，明确：

- 哪些能被用户直接触发
- 哪些只能内部触发
- 内部触发的前置条件是什么

这能显著降低后续规则漂移。

#### 8. 为 scripts 建立能力映射表

当前脚本多，但映射关系分散。  
建议补一份 `references/script-capability-map.md`，把“动作 -> 脚本 -> 产物 -> 触发时机”串起来。

这份表会同时服务：

- 维护者理解
- 测试设计
- 未来自动化验证

---

### 第三层 可选增强

#### 9. 增加安装元信息

可以参考对方后期的产品化做法，补充更完整的 skill 元信息，例如：

- `SKILL.toml`
- 更完整的图标资产
- 更明确的安装说明

这不是能力升级，但会提升可分发性和专业度。

#### 10. 增加面向场景的入口索引

可以补一份“场景入口”文档，把常见使用场景映射到动作：

- 新目标启动
- 中途推进卡住
- 已完成待验收
- 知识补写
- 复盘与原则沉淀

这会比纯动作列表更适合新用户。

#### 11. 为 wiki 自动维护能力建立独立专题

当前 wiki 相关脚本能力已经很强，建议未来单独整理成：

- `references/wiki-automation.md`

避免 `SKILL.md` 和 `README.md` 承担过多解释责任。

---

## 六、建议的实施顺序

### Phase 1

- 新建 `actions/`
- 抽出 8 个公开动作文档
- 精简顶层 `SKILL.md`

### Phase 2

- 新建 `assets/examples/`
- 补 6 个动作示例
- 增加文档一致性测试

### Phase 3

- 新增动作级模板
- 收缩 `README.md`
- 增加脚本能力映射

### Phase 4

- 增加安装元信息
- 增加场景入口索引
- 拆出 wiki 自动化专题

---

## 七、最终建议

对 HeroAgent 来说，最应该“取其精华”的不是 `helloagents_v1` 的大而全规则，而是这三点：

- 用目录结构表达心智模型
- 用模板文件承接稳定产物
- 用示例文件承接行为样例

最不应该照搬的是：

- 过重的顶层主文档
- 过多的阶段分支堆叠
- 让入口层承担太多执行细节

一句话总结：

**HeroAgent 下一步不是变得更像 HelloAGENTS，而是借用它的模块化外壳，把自己已经很强的目标推进与知识沉淀能力组织得更清楚。**
